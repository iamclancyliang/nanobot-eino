func (a *Agent) ChatStream(ctx context.Context, sessionID, input string) {
  a.ensureMCPConnected(ctx)

  taskCtx, taskCancel := context.WithCancel(ctx)
  a.activeTasks.Store(sessionID, taskCancel)

  sess := a.sessions.GetOrCreate(sessionID)

组件之间如何对话

消息总线、会话管理和每会话独立 Worker 模式

机场塔台

想象一座机场塔台。飞机（消息）从不同航空公司（通道）到达。塔台不负责驾驶飞机 —— 它只负责把飞机引导到正确的跑道。如果跑道繁忙，飞机就排队。如果有飞机要起飞，塔台就放行。

这就是发布/订阅消息总线做的事。通道发布入站消息。Agent 订阅并处理它们。Agent 发布出站响应。通道订阅并投递给用户。

ℹ️
为什么不直接调用？

总线让组件保持解耦。飞书通道不导入 Agent 包，Agent 也不导入飞书包。这让测试变得简单（模拟总线即可），添加新通道也轻而易举（只需向总线发布消息）。

每个对话一条专属车道

这里有个巧妙的设计：RunInboundLoop 为每个会话创建一个独立的 Worker 协程。这意味着：

➕

跨会话并行

用户 A 和用户 B 同时被服务，无需排队等待。

🔒

会话内串行

用户 A 的消息按顺序处理，不会出现竞态条件。

🔄

错误自动重试

如果遇到临时故障，等待 2 秒后自动重试。

💡
生产者-消费者模式

这种「每个消费者一个队列」的方式是经典的并发模式。当你让 AI 「同时处理多个用户但保证每个用户的请求按顺序执行」时，它应该用的就是这个模式。

会话 Worker 模式

下面是 RunInboundLoop 创建每会话 Worker 的代码：

}

    ch: make(chan bus.InboundMessage, 16),

if !loaded {

工具安全：包装器

每个工具在 AI 使用之前都会被一层安全层包裹。包装器做两件事：截断（保持结果在 16KB 以内）和进度上报（告诉界面正在发生什么）：

CODE 
func WrapTools(
  inner []tool.InvokableTool,
  maxChars int,
  onProgress ToolProgressFunc,
) []tool.InvokableTool {
  out := make([]tool.InvokableTool, len(inner))
  for i, t := range inner {
    out[i] = &wrappedTool{
      inner:      t,
      maxChars:   maxChars,
      onProgress: onProgress,
    }
  }
  return out
}
通俗解释

接收一组原始工具...

一个最大字符数限制（默认 16,384）...

和一个进度上报函数...

 

创建一个同等大小的新列表...

遍历原始列表中的每个工具...

把它包起来：内核不变，但外面加了大小限制和进度通知

 

 

返回包装后的版本（Agent 使用这些而非原始的）

💡

在不修改工具本身的情况下给它添加行为（包裹一层），这叫「装饰器模式」。就像给手机套一个保护壳 —— 还是同一部手机，但现在有了保护。构建 AI 智能体时，一定要给工具加上安全限制，以防止一个糟糕的 API 返回结果撑爆大模型的上下文。

整合算法

下面是实际的决策逻辑：

CODE 
func (c *MemoryConsolidator) MaybeConsolidateByTokens(
  ctx context.Context, s *session.Session,
) {
  target := c.contextWindowTokens / 2
  estimated := c.estimatePromptTokens(s)
  if estimated <= target { return }

  for round := range maxConsolidationRounds {
    if estimated <= target { return }
    endIdx, _, found := c.PickConsolidationBoundary(
      s, max(1, estimated-target),
    )
    if !found { return }
    chunk := s.Messages[s.LastConsolidated:endIdx]
    c.ConsolidateMessages(ctx, chunk)
  }
}
通俗解释

可能需要归档旧消息以释放空间...

 

 

目标：把提示词控制在上下文窗口的一半以内

粗略估算当前提示词用了多少 token（字符数 ÷ 3）

如果在限制以内，什么都不做，直接退出

 

最多尝试 5 轮整合...

如果已经在限制以内了，停止

找一个安全的切割点（在用户消息的边界上）

归档足够的 token 让我们回到限制以内

如果找不到安全边界，放弃

取出从「上次归档位置」到切割点的旧消息

把它们发给大模型总结，保存到 MEMORY.md + HISTORY.md

懒加载：需要时才连接

MCP（外部工具）服务器不在启动时连接 —— 那样会拖慢启动速度。相反，它们在第一条真实消息时才懒加载连接。就像一台自动咖啡机，只有真的有人点咖啡时才开始加热咖啡豆。

子智能体：任务委派

有时候 AI 需要做一些耗时很长的事情 —— 比如在 20 个网页上做调研。为了不阻塞对话，它会派发一个子智能体 —— 一个拥有自己 AI 大脑的后台工作者。

就像 CEO 把调研任务委派给助理。CEO 继续开会，助理在后台工作。助理完成后，把报告放在 CEO 桌上。

ℹ️
安全防护

子智能体的工具权限受限 —— 不能发送消息，也不能连接 MCP。这防止了无限循环：一个子智能体派发另一个子智能体、发送消息又触发更多智能体...

定时任务

你可以告诉助手"每周一上午9点提醒我回顾本周目标"，它真的能做到。定时任务服务支持三种调度类型：

at
一次性触发

"今天下午3点" —— 执行一次就移除。底层使用 Go 的 time.AfterFunc。

every
间隔执行

"每30分钟" —— 按固定间隔重复执行。内部转换为 cron 表达式。

cron
Cron 表达式

"0 9 * * 1" —— 完整的 cron 语法，用于复杂调度。由 robfig/cron 驱动。
