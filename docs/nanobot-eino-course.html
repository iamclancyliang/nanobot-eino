You Send a Message. What Happens Next?

Tracing a message from your Feishu chat all the way to the AI and back

Meet Nanobot-Eino

Imagine you have a personal assistant that lives inside your Feishu chat. You type a question, and it answers. You ask it to search the web, and it does. You tell it to remind you tomorrow at 9am, and it schedules it.

That's Nanobot-Eino. It's a full AI assistant built in Go that connects to large language models like LLMs, has persistent memory, and can use tools to interact with the world.

💡
Why This Matters for You

Understanding how an AI agent works helps you build better ones. When your AI assistant gets stuck in a loop or forgets context, you'll know exactly where to look.

The Brain: ChatStream

This is the core function that processes every message. Let's read it together:

CODE 
func (a *Agent) ChatStream(ctx context.Context, sessionID, input string) {
  a.ensureMCPConnected(ctx)

  taskCtx, taskCancel := context.WithCancel(ctx)
  a.activeTasks.Store(sessionID, taskCancel)

  sess := a.sessions.GetOrCreate(sessionID)
  a.consolidator.MaybeConsolidateByTokens(taskCtx, sess)

  history := sess.GetHistory(0)
  messages, _ := a.buildMessages(taskCtx, history, input)

  stream, _ := currentAgent.Stream(taskCtx, messages)
}
PLAIN ENGLISH

This is the main "process a message" function on the Agent...

First, connect any external tool servers (only happens once, lazily)

Create a "cancel button" for this task — so the user can type /stop to abort

Store that cancel button, linked to this conversation

Find or create the conversation history for this user

If the conversation is getting too long, archive old parts to save memory

Get the recent conversation history

Package up: system instructions + memory + history + the new message

Send it all to the AI and start streaming the response back

💡
Separation of Concerns

This pattern — giving each component its own folder and clear responsibility — is called "separation of concerns." It means you can change how memory works without touching the agent, or swap Feishu for Slack without rewriting the brain. When telling AI to build something, say "keep the X logic in its own package" — it'll generate cleaner code.

How the Pieces Talk to Each Other

The MessageBus, sessions, and the per-session worker pattern

The Airport Control Tower

Imagine an airport control tower. Planes (messages) arrive from different airlines (channels). The tower doesn't fly the planes — it routes them to the right runway. If a runway is busy, planes queue up. If a plane needs to depart, the tower clears it.

That's what the pub/sub MessageBus does. Channels publish inbound messages. The agent subscribes and processes them. The agent publishes outbound responses. Channels subscribe and deliver them.

ℹ️
Why Not Direct Calls?

The bus keeps components decoupled. The Feishu channel doesn't import the Agent package, and the Agent doesn't import Feishu. This makes testing easy (mock the bus) and adding channels trivial (just publish to the bus).

One Lane Per Conversation

Here's a clever trick: the RunInboundLoop creates one worker goroutine per session. This means:

➕

Parallel Across Sessions

User A and User B get served at the same time. No waiting in line.

🔒

Serial Within a Session

User A's messages are processed in order. No race conditions.

🔄

Auto-Retry on Errors

If something fails transiently, it waits 2 seconds and tries again.

💡
The Producer-Consumer Pattern

This "one queue per consumer" approach is a classic concurrency pattern. When you tell an AI to "handle multiple users at once but keep each user's requests in order," this is the pattern it should reach for.

The Session Worker Pattern

Here's how RunInboundLoop creates per-session workers:

CODE 
type sessionQueue struct {
  ch chan bus.InboundMessage
}

// For each incoming message...
key := msg.Channel + ":" + msg.ChatID
val, loaded := sessions.LoadOrStore(
  key, &sessionQueue{
    ch: make(chan bus.InboundMessage, 16),
  },
)
q := val.(*sessionQueue)

if !loaded {
  wg.Add(1)
  go worker(q)  // new goroutine!
}
q.ch <- msg  // enqueue

Tool Safety: The Wrapper

Every tool is wrapped in a safety layer before the AI uses it. The wrapper does two things: truncation (keep results under 16KB) and progress reporting (tell the UI what's happening):

CODE 
func WrapTools(
  inner []tool.InvokableTool,
  maxChars int,
  onProgress ToolProgressFunc,
) []tool.InvokableTool {
  out := make([]tool.InvokableTool, len(inner))
  for i, t := range inner {
    out[i] = &wrappedTool{
      inner:      t,
      maxChars:   maxChars,
      onProgress: onProgress,
    }
  }
  return out
}
PLAIN ENGLISH

Take a list of raw tools...

a maximum character limit (16,384 by default)...

and a progress reporting function...

 

Make a new list the same size...

For each tool in the original list...

Wrap it: same tool inside, but with a size limit and progress notifications added on top

 

 

Return the wrapped versions (the agent uses these instead of the raw ones)

💡
The Decorator Pattern

Wrapping a tool to add behavior (without changing it) is called the "decorator pattern." It's like putting a phone in a case — same phone, but now it's protected. When building AI agents, always wrap tools with safety limits so one bad API response can't flood the LLM's context.

The Clever Engineering Tricks

Memory consolidation, lazy loading, and graceful shutdown

The Notebook and the Journal

Imagine you keep two notebooks. One is a cheat sheet — key facts you always need (birthdays, allergies, preferences). The other is a daily journal — a timestamped log of everything that happened. When your cheat sheet gets updated, old facts get replaced. The journal only grows.

That's Nanobot's two-layer persistent memory:

📓

MEMORY.md

Long-term facts. Overwritten each time. "User prefers dark mode. User lives in Beijing."

📜

HISTORY.md

Append-only log. Timestamped entries. "[2024-03-15 14:30] Discussed Go patterns. User prefers channel-based concurrency."

The Consolidation Algorithm

Here's the actual decision logic:

CODE 
func (c *MemoryConsolidator) MaybeConsolidateByTokens(
  ctx context.Context, s *session.Session,
) {
  target := c.contextWindowTokens / 2
  estimated := c.estimatePromptTokens(s)
  if estimated <= target { return }

  for round := range maxConsolidationRounds {
    if estimated <= target { return }
    endIdx, _, found := c.PickConsolidationBoundary(
      s, max(1, estimated-target),
    )
    if !found { return }
    chunk := s.Messages[s.LastConsolidated:endIdx]
    c.ConsolidateMessages(ctx, chunk)
  }
}
PLAIN ENGLISH

Maybe archive old messages to free up space...

 

 

Our goal: keep the prompt under half the context window

Roughly estimate how many tokens the full prompt uses (chars ÷ 3)

If we're under the limit, do nothing and exit

 

Try up to 5 rounds of consolidation...

If we're now under the limit, stop

Find a safe cut point (at a user message boundary)

so we archive enough tokens to get under the limit

If no safe boundary exists, give up

Grab the old messages from "last archived" to the cut point

Send them to the LLM to summarize and save to MEMORY.md + HISTORY.md

Lazy Loading: Connect Only When Needed

MCP (external tool) servers aren't connected at startup — that would slow down boot time. Instead, they connect lazily on the first real message. Think of a vending machine that only warms up the coffee beans when someone actually orders coffee.

CODE 
a.mcpOnce.Do(func() {
  for _, cfg := range a.mcpConfigs {
    mcpTools, err := tools.ConnectMCPServer(ctx, cfg)
    if err != nil {
      logAgent.Warn("MCP connection failed", "server", cfg.Name)
      continue
    }
    mcpInvokableTools = append(mcpInvokableTools, mcpTools...)
  }
  // Rebuild the agent with the new tools
  a.reactAgent = a.newReactAgent(ctx, newBaseTools)
})
PLAIN ENGLISH

Do this exactly once (even if called from multiple goroutines)...

For each external tool server in the config...

Try to connect to it

If connection fails...

Log a warning but keep going (don't crash)

Skip this server and try the next one

Add the new tools to our collection

 

Rebuild the AI reasoning engine with the expanded tool set

Subagents: Delegation

Sometimes the AI needs to do something that takes a long time — like researching a topic across 20 web pages. Instead of blocking the conversation, it spawns a subagent — a background worker with its own AI brain.

Think of a CEO who delegates a research task to an intern. The CEO keeps taking meetings while the intern works. When the intern finishes, they leave the report on the CEO's desk.

ℹ️
Safety Guard

Subagents have limited tool access — no message sending and no MCP connections. This prevents infinite loops where a subagent spawns another subagent that sends messages that trigger more agents.

Scheduled Tasks

You can tell the assistant "remind me every Monday at 9am to review my weekly goals" and it actually works. The cron service supports three schedule types:

at
One-time trigger

"At 3pm today" — runs once, then removed. Uses Go's time.AfterFunc under the hood.

every
Interval-based

"Every 30 minutes" — repeats at fixed intervals. Converted to a cron expression internally.

cron
Cron expression

"0 9 * * 1" — full cron syntax for complex schedules. Powered by robfig/cron.
